Server-side feature requests for ANT Neuro EDX service

Source of requests:
- Live BrainFlow testing against EE511 on localhost:3390
- Explicit EDX board id path (BoardIds.ANT_NEURO_EE_511_EDX_BOARD = 81)
- Generic EDX joker path (BoardIds.ANT_NEURO_EDX_BOARD = 67, master_board = 51)

Summary:
- Core acquisition works.
- Main remaining issues are in server-side lifecycle handling, cleanup, and diagnostics.

Requested improvements:

1. Fix release/dispose null-reference errors
- Problem:
  - During stop/release, the server often returns:
    - "Object reference not set to an instance of an object."
- Observed impact:
  - BrainFlow must recover with a direct-dispose fallback.
  - Sessions still succeed, but cleanup is noisy and fragile.
- Request:
  - Make idle/dispose paths null-safe and idempotent.
  - If the stream has already been cancelled, release/dispose should still succeed cleanly.

2. Fix amplifier ownership cleanup timing
- Problem:
  - New sessions sometimes fail with:
    - "Amplifier in use: EE511-010010-001746"
- Observed impact:
  - Back-to-back tests can fail even though the prior session has already stopped.
- Request:
  - Release amplifier ownership immediately and reliably when a session ends.
  - Ensure ownership is cleared before returning success from cleanup APIs.

3. Improve create-device state handling
- Problem:
  - Controller_CreateDevice can fail with wrong-state or object-reference exceptions.
- Observed impact:
  - Port is reachable and device is present, but session creation still fails.
- Request:
  - Normalize device state before create.
  - Return a stable, explicit error when the amplifier is busy or not disposable.
  - Avoid internal exception-based failures.

4. Make cleanup APIs idempotent
- Problem:
  - Repeated or late cleanup calls can trigger server errors after partial teardown.
- Request:
  - Calling stop/idle/dispose more than once should be safe.
  - Server should treat already-stopped and already-disposed cases as success.

5. Eliminate transient startup "sample loss" error reporting
- Problem:
  - Server sometimes reports:
    - "Sample loss because error: sample loss"
  - BrainFlow recovers automatically and steady-state throughput is fine.
- Request:
  - Do not surface stale startup-buffer conditions as hard errors.
  - Clear transition buffers before the first frame of a new stream.
  - Distinguish true sustained sample loss from startup transients.

6. Return clearer machine-readable error categories
- Problem:
  - Multiple different failure modes arrive as generic gRPC unknown/internal-style errors.
- Request:
  - Differentiate at least:
    - amplifier busy
    - invalid state
    - already disposed
    - timeout
    - transport failure
  - Include a stable error code in addition to the message string.

7. Improve device discovery metadata
- Problem:
  - Device model fields can be sparse, and clients may need to infer model/family from key/serial.
- Request:
  - Return explicit model/family/capacity metadata for each discovered amplifier.
  - Include a stable amplifier type/model field suitable for client matching.

8. Improve impedance mode semantics
- Problem:
  - In some runs, impedance values can contain sentinel-like extremes that look like "not available" placeholders.
- Request:
  - Define and document sentinel values for invalid impedance readings.
  - Prefer explicit invalid flags over overloaded numeric extremes.
  - Ensure all impedance rows are populated consistently when impedance mode is active.

9. Add server-side marker/trigger diagnostics
- Problem:
  - Trigger output configuration succeeds, but marker reception depends on loopback and is hard to diagnose from the server side.
- Request:
  - Expose trigger state and marker input diagnostics in a queryable API.
  - Add counters for emitted triggers, received marker edges, and last marker timestamp/code.

10. Add cleanup and ownership logging on the server
- Problem:
  - Current logs make it hard to tell which internal object still owns the amplifier after a failed cleanup.
- Request:
  - Log:
    - amplifier handle creation
    - ownership acquisition/release
    - mode transitions
    - stop/idle/dispose entry and exit
    - whether cleanup completed normally or via fallback

11. Support a "force release stale handle" operation
- Problem:
  - When a previous session dies unexpectedly, the device can remain blocked as "in use".
- Request:
  - Add an explicit admin/recovery API to release stale amplifier ownership safely.

12. Clarify timing guarantees around mode transitions
- Problem:
  - EEG <-> impedance transitions work, but transition timing is not obvious to clients.
- Request:
  - Document when the first valid frame can be expected after SetMode.
  - Guarantee when old buffered frames are discarded.

Requested outcome:
- No null-reference errors during cleanup.
- No spurious "amplifier in use" after successful release.
- No hard startup sample-loss error for healthy streams.
- Stable create/stop/release behavior across repeated sessions.
